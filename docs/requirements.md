# 要件定義書 — Lincoln Price Reflected

## 1. プロジェクト概要

### 1.1 目的
TL-Lincoln（旅館・ホテル向け宿泊管理システム）に対して、Excel ファイルから取得した料金ランク情報を自動的に反映し、反映結果を突合検証するツールを開発する。

### 1.2 解決する課題
- Excel の料金変動案を手作業で Lincoln に入力する際のヒューマンエラー
- 複数施設・複数カレンダーへの一括反映の工数
- 反映結果の目視確認に依存した品質担保

### 1.3 成功基準
**処理C（出力→突合検証）で、Excel 入力値と Lincoln 出力値が完全一致すること。**
1件でも不一致があれば FAILED で停止する。

---

## 2. ステークホルダー

| 役割 | 説明 |
|------|------|
| 運用者 | Windows PC で headful Playwright を実行し、2FA 入力を担当 |
| 管理者 | 施設マスタ・エイリアスの管理、ジョブ結果の確認 |

## 3. 機能要件

### 3.1 処理フロー全体像

```
Excel Upload → 処理0 → 処理A → 処理B → 処理C → 完了/失敗
```

| 処理 | 名称 | Lincoln画面 | 概要 |
|------|------|-------------|------|
| 処理0 | コピー元カレンダーへランク反映 | カレンダー詳細 | Excel の料金ランクを DOM 更新で注入→保存 |
| 処理A | コピー元カレンダー設定 | 6800 detail | 施設ID一致チェック（安全弁） |
| 処理B | 料金ランク一括設定 | 5050 | コピー→送信継続を繰り返し |
| 処理C | 出力→突合検証 | 5070 | 出力データと Excel を比較、完全一致判定 |

### 3.2 処理0 — コピー元カレンダーへランク反映

#### 入力
- パース済み Excel データ（日付×部屋タイプのランクマトリクス）

#### 処理
1. Lincoln のカレンダー詳細画面を開く
2. `page.evaluate` で DOM を直接更新（lincoln リポジトリのロジックを移植）
   - 各日付セルの `inputPriceRankCd`, `inputPriceRankNm`, `inputRankStyleText` を更新
3. 保存/送信ボタンをクリック（セレクタ TBD）
4. 保存成功を検知

#### 出力
- 成功: ステップ完了をジョブに記録
- 失敗: スクリーンショット・HTML・ネットワークログを artifacts に保存

#### 参考
- 既存ツール: https://lincoln-bay.vercel.app/
- 参考リポジトリ: https://github.com/Travel-Connect/lincoln

### 3.3 処理A — コピー元カレンダー設定（施設ID一致チェック）

#### 入力
- 対象施設ID（Excel ファイル名 or UI選択から特定）

#### 処理
1. Lincoln 6800 detail 画面で `dl.g_header_id dd` から施設IDを取得
2. 期待する施設IDと比較
3. **不一致の場合は即座に停止（安全弁）**

#### 出力
- 一致: 処理B へ進む
- 不一致: FAILED で停止、不一致情報をログ出力

### 3.4 処理B — 料金ランク一括設定（カレンダー）

#### 入力
- 処理0で反映済みのコピー元カレンダー情報

#### 処理
1. 5050 画面でコピーボタン `a[onclick="doCopy()"]` をクリック
2. 送信継続ボタン `a[onclick="doSend(true);"]` をクリック
3. 全対象カレンダーに対して 1-2 を繰り返し
4. 各操作後にレスポンスを確認

#### 出力
- 成功: 全カレンダーへの反映完了
- 失敗: リトライ3回→停止

### 3.5 処理C — 出力→突合検証

#### 入力
- 処理B完了後のLincoln データ
- 元の Excel データ

#### 処理
1. 5070 画面で期間を設定（3ヶ月以内制約あり）
2. プラングループを選択
3. 「ランクのみ出力」トグル（セレクタ TBD）を有効化
4. 出力ボタン `a[onclick="doOutput();"]` をクリック
5. 出力された CSV/データと Excel 入力データを突合

#### 出力
- **完全一致**: SUCCESS — ジョブ完了
- **不一致あり**: FAILED — 不一致詳細をログ/CSV/JSON で保存し停止

#### 制約
- 出力期間は最大3ヶ月。対象期間が3ヶ月を超える場合は分割して出力・検証する。

### 3.6 Excel 入力パース

#### ファイル名規約
```
【{施設名}様】料金変動案_{YYYYMMDD}[_自動生成].xlsx
```

#### パース仕様
- シート名: 固定（設定可能）
- データ起点: C4 セル
- 範囲: 右方向・下方向ともに空白セルで終端検出
- 出力: 日付×部屋タイプのランクマトリクス

#### 施設名抽出
- ファイル名の `【...様】` から候補を抽出
- Supabase の施設マスタ（aliases 含む）と照合
- 照合失敗時は UI で施設選択を促す

### 3.7 A/B（単泊/連泊）判定
- Excel 内では判定不可
- ユーザーが UI 上で手動紐づけする
- 紐づけ結果はジョブデータに保存

## 4. 非機能要件

### 4.1 安全性（最優先）

| 要件 | 対策 |
|------|------|
| 誤反映防止 | 実行前にサマリ表示（変更件数・期間・対象施設・カレンダー・プラングループ/プラン）→ユーザーOKで実行 |
| 検証 | 処理Cで完全一致検証。差分表UIは作らない |
| 施設ID検証 | 処理Aで施設ID一致チェック必須 |
| 不一致時 | FAILED で即停止。不一致詳細はログ/CSV/JSONで保存 |

### 4.2 再開可能性（Resume）
- 各ステップ完了ごとに `jobs.last_completed_step` を更新
- 再実行時は `last_completed_step` の次から継続
- ステップ: `PARSE → STEP0 → STEPA → STEPB → STEPC → DONE`

### 4.3 エラーハンドリング
- 各操作でリトライ N 回（デフォルト3）
- リトライ超過で停止
- 失敗時は artifacts（スクリーンショット・HTML・ネットワークログ）を保存

### 4.4 認証
- Lincoln ログインURL: `https://www.tl-lincoln.net/accomodation/Ascsc1000InitAction.do`
- ID/PW は環境変数で管理（DB に平文保存しない）
- 2FA: 検出時にユーザー入力を促し、入力完了を検知して継続
- 施設切替: 同一アカウントでセレクタ操作（セレクタ TBD）

### 4.5 セレクタ管理
- 全セレクタは `config/selectors.json` に集約
- コード内にセレクタ文字列を直書き禁止
- TBD セレクタはガード付きで安全に停止

## 7. 制約事項

1. Playwright は Windows 作業者PC で headful 実行（ヘッドレス不可：2FA対応のため）
2. 処理Cの出力期間は最大3ヶ月
3. 差分表 UI は作らない（ログ/CSV/JSON のみ）
4. セレクタはコード直書き禁止（config/selectors.json に集約）
5. TBD セレクタが未設定の処理は実行せずガードで停止
