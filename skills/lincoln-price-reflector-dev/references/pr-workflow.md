# PR 分割テンプレ

## 基本方針

- **1PR = 1責務**（機能単位 or レイヤー単位）
- 同一ファイルを複数 PR で同時編集しない
- 依存関係に沿って順番にマージ
- 各 PR にはテスト or 動作確認手順を含める
- PR 作成はスラッシュコマンド (`/commit`, GitHub CLI) を使用

## PR テンプレート

```markdown
## Summary
- <1-3 行の変更概要>

## 変更内容
- <具体的なファイル・機能の変更点>

## テスト・確認手順
- [ ] <確認項目 1>
- [ ] <確認項目 2>

## 依存 PR
- #<依存する PR 番号> (マージ済み / 未マージ)

## チェックリスト
- [ ] `C:\lincolnpricereflected` 配下のみに変更
- [ ] セレクタは `config/selectors.json` から読み込み
- [ ] secrets がコード/ログに含まれていない
- [ ] 型チェック / lint がパス
```

## PR 分割の例

### 良い例

| PR | 責務 |
|----|------|
| PR#14 | セレクタローダ + TBD ガード |
| PR#15 | リトライユーティリティ |
| PR#16 | artifacts 保存ユーティリティ |

→ それぞれ独立したユーティリティ。個別にレビュー・テスト可能。

### 悪い例

| PR | 問題 |
|----|------|
| PR#X | セレクタローダ + リトライ + artifacts + ジョブ状態管理 |

→ 責務が混在。レビューが困難で、一部の問題が全体をブロックする。

## ブランチ命名規則

```
<type>/<short-description>
```

| type | 用途 |
|------|------|
| feat/ | 新機能 |
| fix/ | バグ修正 |
| refactor/ | リファクタリング |
| docs/ | ドキュメント更新 |
| chore/ | 設定・依存関係等 |

例: `feat/selector-loader`, `fix/retry-backoff`, `docs/selectors-catalog`

## WBS との対応

PR 番号は `docs/wbs.md` の WBS と対応させる。
Phase・PR 番号を参照して依存関係を確認すること。
